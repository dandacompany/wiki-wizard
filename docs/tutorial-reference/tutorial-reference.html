OVERVIEW

개요 — 두 표면 모델

oh-my-wiki는 Andrej Karpathy의 "LLM Wiki" 워크플로를 구현합니다. 모든 소스는 raw 스냅샷, 요약 페이지, 10–15개의 엔티티·개념 페이지 터치로 이어지고, 쿼리는 구조화된 위키에서 출처를 인용해 답합니다. host-universal — 하나의 SKILL.md가 Claude Code · Codex · Gemini에서 동일하게 동작합니다. 인터페이스는 정확히 두 개입니다.

🎯omw를 두 표면으로 쓴다 — 결정론 CLI와 세션 내 스킬.
📏 PRINCIPLES
1페르소나는 제안, 결정론 명령이 실행 (propose → confirm → execute).
2어떤 AI 호스트(Claude Code · Codex · Gemini)에서도 동일한 SKILL.md, 동일한 트리거 문구.
3모든 파일 변경은 감사 가능 — 추론은 투명하게, 출력은 결정론으로.

A

설치 · 설정

세 가지 설치 경로 중 환경에 맞는 하나를 고릅니다. 설치 후 omw setup이 레지스트리와 첫 vault를 만들고, omw status·omw doctor가 연결 상태를 검증합니다.

PATH A — Skills CLI (Claude Code 권장)
bash
skills add dandacompany/oh-my-wiki@oh-my-wiki -g -y --copy -a claude-code
스킬을 ~/.claude/skills/에 설치하고 oh-my-wiki와 단축 별칭 omw를 함께 등록합니다.
PATH B — Claude Code 플러그인 마켓플레이스
claude code
/plugin marketplace add dandacompany/oh-my-wiki
/plugin install oh-my-wiki@oh-my-wiki-marketplace
이후 업데이트는 /plugin marketplace update oh-my-wiki-marketplace로 진행합니다.
PATH C — git clone + install script (Codex CLI · 개발자)
bash
git clone https://github.com/dandacompany/oh-my-wiki
cd oh-my-wiki
bash bin/install.sh
인스톨러는 Python 3.10+ 확인, pip install -e ".", ~/.claude/skills/oh-my-wiki·omw symlink 생성(멱등성), pytest 검증을 수행합니다. --dev(테스트 의존성), --force(프롬프트 없이 교체), --no-test, --skills-dir <path> 플래그를 받습니다.
설정 마법사 — omw setup
terminal
omw setup
섹션을 순서대로 진행하는 대화형 마법사입니다. 섹션 인자를 주면 한 섹션만 다시 조정합니다: vault · hosts · search · serve · personas · tts · import. --noninteractive는 플래그·기본값만으로 프롬프트 없이 생성합니다.
상태 확인 — omw status (깨끗한 머신)
json
{
  "vault_count": 0,
  "active": null,
  "needs": "setup",
  "vaults": []
}
needs는 setup(첫 vault 필요) · select(vault 선택 필요) · migrate(레거시 data/registry.db 감지) · op(준비 완료) 중 하나입니다. 소스 트리에서 실행하면 data/registry.db 때문에 needs: "migrate"가 표시되며 이는 개발 트리에서만 나타납니다.
설치 건강 검사 — omw doctor
output
omw home:   ~/.omw  ok
registry:   ~/.omw/registry.db  ok
  * demo (wiki/markdown) ~/.omw/vaults/demo
doctor는 각 컴포넌트를 찾으면 ok를, 없으면 무엇이 빠졌는지 보고합니다. omw setup 전이라면 missing (run: omw setup)으로 표시됩니다.
★ OMW_HOME — 레지스트리는 $OMW_HOME/registry.db에 저장됩니다(기본값 ~/.omw/registry.db). 설정·토큰은 ~/.omw/config.yaml과 ~/.omw/.env(권한 0600)에 저장됩니다.
  • export OMW_HOME=/path/to/isolated/.omw로 완전히 분리된 wiki 환경을 운영합니다. 테스트·CI·다중 환경 격리에 유용합니다.
B

볼트 · 수집

vault는 하나의 지식 저장소입니다. omw vault 4개 동작으로 생성·전환하고, omw import로 기존 자료를 끌어오며, 세션에서 자연어 ingest로 새 소스를 흡수합니다.

vault 생성 — omw vault create
terminal
omw vault create demo --mode wiki
--mode는 memo(가벼운 노트) 또는 wiki(Karpathy raw/wiki/index/log 패턴). --type은 markdown 또는 obsidian. --location은 global · project · 절대 경로를 받습니다.
성공하면 보이는 것
json
{
  "created": "demo",
  "path": "~/.omw/vaults/demo",
  "mode": "wiki",
  "type": "markdown"
}
목록 · 전환 · 제거
terminal
omw vault list           # 등록된 vault를 JSON으로 (is_active 포함)
omw vault use demo       # 활성 vault 전환
omw vault forget demo    # 레지스트리 행만 제거 — 파일은 보존
vault forget은 디스크 파일을 절대 건드리지 않고 레지스트리 행만 지웁니다. 같은 경로로 다시 create하면 복구됩니다.
기존 자료 가져오기 — omw import
terminal
omw import --source folder   --src-dir ./notes
omw import --source obsidian --src-dir ~/Obsidian/MyVault
omw import --source notion   --notion-id <export-id>
--source는 folder · obsidian · notion. --layer는 가져온 파일을 raw(스냅샷) 또는 wiki 레이어 중 어디에 둘지 정합니다. --vault로 대상 vault를 지정합니다(기본값: 활성 vault).
세션에서 새 소스 흡수 — ingest (자연어)
ai session
ingest this

<붙여넣은 기사·링크·문서 본문>
스킬이 제목·slug·태그·저장 위치를 제안하고, 확인하면 저장합니다. 소스 하나당 raw 스냅샷 1개 + 요약 1개 + 엔티티·개념 페이지 10~15개 터치가 정상입니다.

C

질의 · 검색

vault 내부 검색은 SQLite FTS5(BM25)로 동작합니다. 웹 검색은 별개로 외부 provider를 통합니다. 세 가지 경로 — 세션 내 자연어, 로컬 HTTP API, 외부 웹 검색 — 를 용도에 맞게 구분합니다.

vault 인덱싱 방식
fts5
title + summary + tags + body  →  SQLite FTS5 (BM25)
FTS5 미지원 시  →  토큰 스코어 기반 자동 폴백
세션에서 "내 위키에서 X에 대해 뭐라고 해?"라고 물으면 스킬이 FTS5로 후보를 찾고 LLM이 결과를 재순위 매깁니다(임베딩 없음). 추론은 세션에서, 검색은 결정론으로.
세션 내 자연어 질의 — query (자연어)
ai session
내 위키에서 compounding knowledge에 대해 뭐라고 해?
commands/query.md의 절차로 FTS5 후보를 가져온 뒤 LLM이 재순위 매기고 출처를 인용해 답합니다.
외부 웹 검색 — omw search
terminal
omw search "LLM wiki pattern" --provider brave --limit 5
omw search는 외부 provider(brave · tavily · exa · firecrawl · brightdata)를 통한 웹 검색이며 vault 내부를 검색하지 않습니다. provider 미설정 시 다음을 출력합니다 — error: no search provider configured — run `omw setup search`.
provider 설정 — omw setup search
terminal
omw setup search
★ autoresearch — 세션에서 autoresearch <질문>이라고 말하면 질문을 주장 단위로 분해하고, 주장별 최대 3라운드 웹 검색 후 confidence 태그를 부여하며, synthesis 초안을 작성해 저장 전 확인을 요청합니다. → wiki/syntheses/<slug>.md
D

작성 — 페르소나 · 팀

작성·검증·유지 추론은 16종 writing persona가 담당합니다. 세션에서 자연어로 호출하면 입력 내용에 따라 스킬이 적절한 persona로 라우팅합니다. 여러 persona를 묶은 9종 팀 템플릿으로 파이프라인을 한 번에 돌립니다.

★ 공통 모델 — propose → confirm → execute — 모든 persona는 파일을 읽고, 제안 초안을 작성하고, 무엇이 바뀔지 보여준 다음에야 작성합니다. 사람의 확인 없이는 어떤 파일도 변경되지 않습니다.
  • input_kinds는 각 persona가 받는 입력(text · file · vault_page) 종류입니다.
  • output_kind는 결과가 어디로 가는지를 나타냅니다 — stdout(세션 출력) · new_page(새 페이지) · sibling_file·sibling_suffix(형제 파일) · inplace(제자리 편집).
Researcher — 세션에서:
ai session
autoresearch how does the LLM Wiki pattern compare to Zettelkasten?
다라운드 웹 리서치로 출처를 가중하고, 인라인 인용·confidence 태그가 붙은 구조화 페이지 초안을 만듭니다. → new_page
Fact-checker — 세션에서:
ai session
fact-check wiki/concepts/llm-wiki.md
초안을 원자 단위 주장으로 분해해 웹 검색으로 검증하고, 판정 표(supported · contradicted · partial · unverifiable)를 <page>.factcheck.md에 작성합니다. → sibling_suffix
Polisher / Translator / Summarizer — 세션에서:
ai session
polish this            # 번역투 제거·문체 정리 (제자리, .trash 백업)
translate this to Korean   # frontmatter·코드블록 보존, 형제 파일 생성
summarize this         # 한 줄 / 한 단락 / 상세 3단 JSON
Polisher는 --lang ko에서 korean-prose-polish 규칙을 따릅니다. Translator·Fact-checker는 웹 검색 도구(mcp__brightdata__search_engine)를 사용합니다.
Wiki-auditor / Wiki-librarian / Curator — 세션에서:
ai session
check my wiki for contradictions   # auditor: 건강 진단
build a glossary for my vault      # terminology-manager
curate my wiki                     # index 재정렬 제안
auditor는 무엇이 아픈지 진단하고, librarian은 어떻게 고칠지 제안하며(교차 링크·고립 해소·병합), curator는 wiki/index.md를 재동기화합니다.

E


유지 — 규약 · 신뢰도 · 링크


vault 건강은 결정론 명령으로 유지합니다. 스키마가 페이지 규약을 정의하고, supersede·review가 신뢰도와 수명을, links·fields가 그래프를, lint가 전체 health를 관리합니다.


페이지 규약 조회 — omw schema list / show
terminal
omw schema list                    # 13개 타입 + 각 타입 요구사항
omw schema show entity             # 한 타입의 유효 스키마
omw schema show entity --vault demo   # vault 오버라이드 적용


13개 타입: article · book · comparison · concept · doc · entity · link · meta · note · paper · summary · synthesis · video. entity만 본문에 ## Summary 섹션을 요구하고, meta는 필수 필드를 면제받습니다.


vault별 스키마 오버라이드
tree
~/.omw/vaults/demo/
└── schemas/
    └── entity.yml   ← 이 vault에서만 내장 entity 스키마를 오버라이드


<vault>/schemas/의 파일이 패키지 루트 schemas/의 내장 기본값보다 우선합니다. 공유 기본값을 건드리지 않고 특정 프로젝트에 커스텀 타입을 추가하거나 필드 규칙을 강화합니다.


신뢰도 대체 — omw supersede
terminal
omw supersede wiki/concepts/old-method.md --by llm-wiki


페이지를 삭제하지 않고 frontmatter에 status: superseded + superseded_by: <slug>를 써서 감사 추적을 보존합니다. 반환 JSON은 {relpath, status, superseded_by}.


리뷰 주기 — omw review done / due
terminal
omw review done wiki/concepts/llm-wiki.md --grade pass --today 2026-06-01
omw review due --today 2026-09-01 --scheduled-only


--grade는 pass 또는 needs-work. 간격은 confidence를 따릅니다 — high → 90일, medium → 30일, low → 7일. due는 {relpath, due, interval_days, confidence} 목록을 반환하며, review: 블록이 없는 페이지는 due: null로 가장 앞에 정렬됩니다. --scheduled-only는 미검토 페이지를 제외합니다.


엔티티 자동 링크 — omw links suggest / link
terminal
omw links suggest                                    # 링크 없는 언급 목록
omw links suggest wiki/concepts/llm-wiki.md          # 한 페이지로 한정
omw links link wiki/concepts/llm-wiki.md --to andrej-karpathy


suggest는 {src_relpath, target_slug, target_relpath, mention, position} 목록을, link는 첫 언급을 [[andrej-karpathy|Andrej Karpathy]]로 제자리에서 재작성합니다. 페이지는 aliases: frontmatter 목록으로 매칭 대상을 늘립니다.


★ 한국어 조사 매칭 — 본문에 안드레이 카르파시가 이 방법을 제안했다.처럼 조사 가가 붙어 있어도, suggest가 이름을 slug와 매칭하고 link는 [[…|안드레이 카르파시]]가 이 방법을 제안했다.로 삽입합니다. 조사는 wikilink 괄호 밖에 남고 표시 텍스트는 조사 없는 표준 이름입니다.


인라인 필드 — omw fields
terminal
omw fields wiki/concepts/llm-wiki.md


본문에 Dataview 문법 key:: value를 넣으면 frontmatter와 함께 파싱·인덱싱됩니다. 반환 JSON은 {relpath, frontmatter, inline}. 관계 키(uses · contradicts · supersedes:: [[B]])는 frontmatter relations:와 동일하게 타입드 엣지 그래프에 반영됩니다.


전체 health 검사 — omw lint
json
{
  "vault_id": 1,
  "vault_path": "~/.omw/vaults/demo",
  "frontmatter_issues": [],
  "drift": { "missing_files": [], "mtime_drift": [] },
  "links": {
    "broken": [], "orphans": [],
    "index_drift": { "missing_from_index": [], "dangling_in_index": [] },
    "contradictions": [], "supersedes": [],
    "superseded_unmarked": [], "link_suggestions": []
  },
  "auto_fix_hints": []
}


frontmatter_issues는 필수 필드 검사, drift는 디스크↔인덱스 불일치, links의 7개 키는 구조 건강(broken · orphans · index_drift · contradictions · supersedes · superseded_unmarked · link_suggestions), auto_fix_hints는 실행 가능한 해결책을 담습니다. --vault로 대상 vault를 지정합니다.

운영


omw serve가 메신저 봇을 위한 로컬 읽기 전용 쿼리 API를 띄웁니다. host export로 어떤 AI 호스트에서도 같은 스킬을 쓰고, OMW_HOME 디렉토리 복사만으로 백업·이전이 끝납니다.


인증 토큰 생성 — omw setup serve
terminal
omw setup serve --generate-token


토큰을 ~/.omw/.env에 OMW_SERVE_TOKEN으로 권한 0600으로 저장합니다.


서버 시작 — omw serve
terminal
omw serve                                        # http://127.0.0.1:8765 (localhost 전용)
omw serve --host 0.0.0.0 --port 9000 --vault demo --limit 5


기본 바인딩은 localhost입니다. 공개·TLS 노출은 리버스 프록시(예: Caddy)로 앞단을 구성합니다 — 서버 자체는 TLS를 종료하지 않습니다. --limit은 응답당 최대 hit 수 상한입니다.


메신저 API — curl
bash
TOKEN=$(grep OMW_SERVE_TOKEN ~/.omw/.env | cut -d= -f2)

# health (인증 불필요)
curl -s http://127.0.0.1:8765/health
# {"status":"ok"}

# query (POST + bearer token)
curl -s -X POST http://127.0.0.1:8765/query \
  -H "Authorization: Bearer $TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"text":"what is attention?","limit":3}'


POST /query는 인증 필요, GET /health는 인증 불필요, GET /query는 405를 반환합니다. 본문 필드 — text(필수) · user · channel(로그 패스스루) · vault · limit.


응답 형식 (200) — 재순위 hit 목록 (LLM 없음)
json
{
  "query": "what is attention?",
  "vault": "ai-research",
  "count": 1,
  "hits": [
    {
      "relpath": "wiki/concepts/attention.md",
      "title": "Attention",
      "summary": "...",
      "tags": ["nlp"],
      "score": 12.3
    }
  ]
}


서버는 답을 합성하지 않고 랭커 결과를 그대로 돌려줍니다. 메신저(Slack · Telegram · Discord) 어댑터는 이 hit을 메시지로 포매팅하는 얇은 webhook입니다. 에러는 모두 JSON — 401(토큰) · 400(본문) · 404(vault) · 409(활성 vault 없음) · 405(메서드) · 500(내부). 전체 형식은 references/messenger-api.md에 있습니다.


host export — 어떤 AI 호스트에서도 동일
terminal
omw setup hosts --host claude,codex,gemini


CLAUDE.md · AGENTS.md(Codex) · GEMINI.md에 동일한 트리거 문구를 export합니다. --base-dir로 instruction 파일 위치를 지정합니다. host-universal — 한 SKILL.md가 모든 호스트에서 같은 방식으로 동작합니다.


★ 백업 · 이전 — 모든 상태가 $OMW_HOME(기본 ~/.omw) 한 곳에 모입니다 — registry.db · config.yaml · .env · vaults/.
  • 이 디렉토리를 복사하면 백업이, 새 머신에서 같은 위치에 두고 OMW_HOME을 가리키면 이전이 끝납니다.
  • SMB 마운트 vault(/Volumes/…)는 cp -a 대신 rsync -rlpt를 사용합니다.
