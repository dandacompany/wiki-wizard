OVERVIEW

두 표면 모델

oh-my-wiki는 Andrej Karpathy의 "LLM Wiki" 워크플로를 구현합니다. 모든 소스는 raw 스냅샷, 요약 페이지, 10–15개의 엔티티·개념 페이지 터치로 이어지고, 쿼리는 구조화된 위키에서 출처를 인용해 답합니다. 인터페이스는 정확히 두 개입니다.

🎯omw를 두 표면으로 쓴다.
📏 PRINCIPLES
1페르소나는 제안, 결정론 명령이 실행 (propose → confirm → execute).
2어떤 AI 호스트(Claude Code · Codex · Gemini)에서도 동일한 SKILL.md, 동일한 트리거 문구.
3모든 파일 변경은 감사 가능 — 추론은 투명하게, 출력은 결정론으로.
🧱 COMPONENTS
⌨️

💬
omw 스킬
세션 내 자연어 추론 — ingest·query·autoresearch·페르소나
STEP 01

설치

환경에 맞는 경로 하나를 골라 설치합니다. 어떤 경로든 끝나면 omw doctor로 연결 상태를 확인합니다.

PATH A — Skills CLI (Claude Code 권장)
bash
skills add dandacompany/oh-my-wiki@oh-my-wiki -g -y --copy -a claude-code
스킬을 ~/.claude/skills/에 설치하고 oh-my-wiki와 단축 별칭 omw 스킬 이름을 함께 등록합니다.
PATH B — Claude Code 플러그인 마켓플레이스
claude code
/plugin marketplace add dandacompany/oh-my-wiki
/plugin install oh-my-wiki@oh-my-wiki-marketplace
이후 업데이트는 /plugin marketplace update oh-my-wiki-marketplace로 진행합니다.
PATH C — git clone + install script (Codex CLI · 개발자)
bash
git clone https://github.com/dandacompany/oh-my-wiki
cd oh-my-wiki
bash bin/install.sh
인스톨러가 Python 3.10+ 확인, pip install -e ".", ~/.claude/skills/oh-my-wiki·omw symlink 생성(멱등성), pytest -q 검증을 수행합니다. 재실행해도 안전하며, --force로 프롬프트 없이 교체합니다.
설치 확인 — omw doctor
terminal
omw doctor
성공하면 보이는 것 (vault가 있을 때)
output
omw home:   ~/.omw  ok
registry:   ~/.omw/registry.db  ok
  * demo (wiki/markdown) ~/.omw/vaults/demo
새 머신에서 omw setup 전이라면 missing (run: omw setup)으로 표시됩니다. doctor는 각 컴포넌트를 찾으면 ok를, 없으면 무엇이 빠졌는지 보고합니다.
STEP 02

첫 위키

설정 마법사로 첫 vault를 만들고, 상태와 lint로 깨끗한 시작점을 확인합니다.

설정 마법사 실행
terminal
omw setup
첫 vault, 검색 provider, TTS, persona를 구성하는 대화형 마법사입니다. 기본값을 그대로 받아들이면 빠르게 시작합니다. 이후 omw setup vault·omw setup personas로 개별 섹션을 다시 조정합니다.
상태 확인 — omw status
terminal
omw status
성공하면 보이는 것 (깨끗한 머신)
json
{
  "vault_count": 0,
  "active": null,
  "needs": "setup",
  "vaults": []
}
needs: "setup"은 깨끗한 머신의 정상 화면입니다. 소스 트리에서 실행하면 data/registry.db 때문에 needs가 "migrate"로 표시되며, 이는 개발 트리에서만 나타납니다.
첫 vault 만들기
terminal
omw vault create demo --mode wiki
성공하면 보이는 것
json
{
  "created": "demo",
  "path": "~/.omw/vaults/demo",
  "mode": "wiki",
  "type": "markdown"
}
활성 상태 확인 — omw vault list
json
[
  {
    "name": "demo",
    "path": "~/.omw/vaults/demo",
    "mode": "wiki",
    "type": "markdown",
    "is_active": true
  }
]
노트 추가 (Claude / Codex / Gemini 세션에서)
ai session
ingest this

Andrej Karpathy calls the LLM Wiki a "compounding knowledge artifact". Every
source gets saved verbatim to raw/, a summary lands at wiki/summaries/, and
the entities and concepts that appeared get their own pages. 10–15 page touches
per ingest is normal.
스킬이 제목, slug, 태그, 저장 위치를 제안합니다. 확인하면 저장됩니다.
lint 검사 — omw lint
terminal
omw lint
성공하면 보이는 것 (문제 없는 vault)
json
{
  "vault_id": 1,
  "vault_path": "~/.omw/vaults/demo",
  "frontmatter_issues": [],
  "drift": { "missing_files": [], "mtime_drift": [] },
  "links": {
    "broken": [],
    "orphans": [],
    "index_drift": { "missing_from_index": [], "dangling_in_index": [] },
    "contradictions": [],
    "supersedes": [],
    "superseded_unmarked": [],
    "link_suggestions": []
  },
  "auto_fix_hints": []
}
★ 읽는 법 — frontmatter_issues: []는 모든 페이지가 필수 필드 검사를 통과했다는 뜻입니다.
  • links 키들(broken·orphans·index_drift·contradictions·supersedes·superseded_unmarked·link_suggestions)은 vault의 구조 건강 상태를 알려줍니다.
  • drift는 디스크에 있지만 인덱스에 없는 파일을 보고합니다.
  • auto_fix_hints는 문제가 발견될 때 실행 가능한 해결 방법을 제시합니다.
STEP 03

페이지 규약 (스키마)

13개 내장 페이지 타입이 각 타입에 필요한 필드와 허용값을 정의합니다. 타입을 나열하고, entity 타입을 자세히 확인합니다.

타입 나열 — omw schema list
terminal
omw schema list
13개 타입
output
article, book, comparison, concept, doc, entity, link, meta, note,
paper, summary, synthesis, video
각 항목은 type, required_fields, required_sections, field_types, allowed_values를 가진 스키마 객체입니다.
entity 타입 상세 — omw schema show entity
terminal
omw schema show entity
성공하면 보이는 것
json
{
  "type": "entity",
  "required_fields": ["title", "date", "type", "tags"],
  "required_sections": ["## Summary"],
  "field_types": {
    "tags": "list",
    "title": "str",
    "date": "str",
    "review": "dict",
    "aliases": "list"
  },
  "allowed_values": {
    "confidence": ["high", "medium", "low"],
    "status": ["draft", "inbox", "processed", "raw", "superseded", "meta"]
  }
}
모든 entity 페이지는 본문에 ## Summary 섹션이 있어야 합니다. confidence는 high·medium·low를, status는 allowed_values 목록값을 허용합니다.
vault별 스키마 오버라이드
tree
~/.omw/vaults/demo/
└── schemas/
    └── entity.yml   ← overrides the built-in entity schema for this vault only
<vault>/schemas/의 파일이 패키지 루트의 내장 schemas/보다 우선합니다. 공유 기본값을 건드리지 않고 특정 프로젝트에 커스텀 타입을 추가하거나 필드 규칙을 강화합니다.
STEP 04

신뢰도와 대체

페이지는 근거의 충분함을 나타내는 confidence(high·medium·low)를 가집니다. 더 나은 페이지로 대체할 때는 삭제하지 않고 superseded로 표시해 감사 추적을 보존합니다.

old-method.md를 llm-wiki로 supersede
terminal
omw supersede wiki/concepts/old-method.md --by llm-wiki
성공하면 보이는 것
json
{
  "relpath": "wiki/concepts/old-method.md",
  "status": "superseded",
  "superseded_by": "llm-wiki"
}
old-method.md에 작성되는 frontmatter
yaml
status: superseded
superseded_by: llm-wiki
omw lint는 본문에 "outdated"·"replaced"로 비공식 설명되어 있지만 이 필드가 없는 페이지를 superseded_unmarked 키 아래에 표시합니다.
STEP 05

리뷰 주기

모든 페이지는 frontmatter의 review: 블록으로 다음 재평가 일정을 지정합니다. 간격은 confidence를 따릅니다. high → 90일, medium → 30일, low → 7일.

llm-wiki.md (high-confidence) review 완료 처리
terminal
omw review done wiki/concepts/llm-wiki.md --grade pass --today 2026-06-01
성공하면 보이는 것
json
{
  "relpath": "wiki/concepts/llm-wiki.md",
  "review": { "last": "2026-06-01", "due": "2026-08-30", "interval_days": 90 }
}
high confidence → 90일 간격 → 만료일 2026-08-30.
review 대상 조회 (미래 날짜 시뮬레이션)
terminal
omw review due --today 2026-09-01
{relpath, due, interval_days, confidence} 목록이 반환됩니다. review: 블록이 없는 페이지는 due: null로 표시되며 가장 앞에 정렬됩니다. 한 번도 검토되지 않았으므로 주의가 필요합니다.
STEP 06

전문 검색 & 메신저 API

vault 검색은 SQLite FTS5(BM25)로 동작합니다. 세션에서 자연어로 묻거나, omw serve로 로컬 읽기 전용 HTTP API를 띄워 POST /query로 가져옵니다.

vault 인덱싱 방식
fts5
title + summary + tags + body  →  SQLite FTS5 (BM25)
FTS5 미지원 시  →  토큰 스코어 기반 자동 폴백
세션에서 "내 위키에서 X에 대해 뭐라고 해?"라고 말하면 스킬이 FTS5로 검색하고 LLM이 결과를 재순위 매깁니다. omw serve는 서버에 LLM 없이 검색만 수행합니다.
인증 토큰 생성 (~/.omw/.env에 OMW_SERVE_TOKEN으로 저장)
terminal
omw setup serve --generate-token
서버 시작 — omw serve
terminal
omw serve
서버는 http://127.0.0.1:8765(localhost 전용)에서 실행됩니다. POST /query는 인증 필요, GET /health는 인증 불필요, GET /query는 405를 반환합니다.
curl — health(인증 없음) + query(POST + bearer)
bash
# health (no auth)
curl -s http://127.0.0.1:8765/health

# query (POST + bearer token)
curl -s -X POST http://127.0.0.1:8765/query \
  -H "Authorization: Bearer $OMW_SERVE_TOKEN" \
  -H "Content-Type: application/json" \
  -d '{"query": "compounding knowledge", "limit": 5}'
전체 요청/응답 JSON 형식과 Slack·Telegram·Discord 어댑터 스케치는 references/messenger-api.md에 있습니다.
★ 구분 — omw search는 별개입니다. 외부 provider(brave·tavily·exa·firecrawl·brightdata)를 통한 웹 검색이며, vault 내부를 검색하지 않습니다. provider 미설정 시 omw setup search를 실행합니다.
STEP 07

엔티티 자동 링크

개념 페이지가 엔티티를 언급하면서 링크를 걸지 않는 경우, oh-my-wiki가 링크 없는 언급을 감지하고 자동으로 wikilink를 삽입합니다.

링크 없는 언급 감지 — omw links suggest
terminal
omw links suggest
성공하면 보이는 것 (2건)
json
[
  {
    "src_relpath": "wiki/concepts/llm-wiki.md",
    "target_slug": "andrej-karpathy",
    "target_relpath": "wiki/entities/andrej-karpathy.md",
    "mention": "Andrej Karpathy",
    "position": 145
  },
  {
    "src_relpath": "wiki/entities/andrej-karpathy.md",
    "target_slug": "llm-wiki",
    "target_relpath": "wiki/concepts/llm-wiki.md",
    "mention": "LLM Wiki",
    "position": 88
  }
]
llm-wiki.md 위치 145에서 "Andrej Karpathy"가, andrej-karpathy.md 위치 88에서 "LLM Wiki"가 wikilink 없이 언급됩니다. 두 경우 모두 vault에 일치하는 페이지가 존재합니다.
링크 삽입 — omw links link
terminal
omw links link wiki/concepts/llm-wiki.md --to andrej-karpathy
성공하면 보이는 것
json
{
  "relpath": "wiki/concepts/llm-wiki.md",
  "target_slug": "andrej-karpathy",
  "mention": "Andrej Karpathy",
  "inserted": "[[andrej-karpathy|Andrej Karpathy]]"
}
해당 언급을 [[andrej-karpathy|Andrej Karpathy]]로 제자리에서 재작성합니다.
★ 한국어 조사 매칭 — 본문에 안드레이 카르파시가 이 방법을 제안했다.처럼 조사 가가 이름에 붙어 있어도, omw links suggest가 안드레이 카르파시가를 안드레이 카르파시 slug와 매칭하고, omw links link는 다음처럼 삽입합니다.
  • [[…|안드레이 카르파시]]가 이 방법을 제안했다.
조사는 wikilink 괄호 밖에 남고, 표시 텍스트는 조사 없는 표준 이름입니다.
STEP 08

인라인 필드

페이지 본문에 Dataview 스타일 인라인 필드 key:: value를 넣으면 frontmatter와 함께 파싱·저장·인덱싱됩니다.

본문 인라인 필드 예시
markdown
owner:: dante
status:: draft
uses:: [[llm-wiki]]
전체 필드 확인 — omw fields
terminal
omw fields wiki/concepts/llm-wiki.md
성공하면 보이는 것
json
{
  "relpath": "wiki/concepts/llm-wiki.md",
  "frontmatter": {
    "title": "LLM Wiki",
    "date": "2026-06-01",
    "type": "concept",
    "tags": ["method"]
  },
  "inline": { "owner": ["dante"], "status": ["draft"] }
}
wikilink를 참조하는 관계 키(uses, contradicts, supersedes)는 frontmatter relations:와 동일하게 타입드 엣지 그래프에 반영됩니다.
STEP 09

페르소나 (세션 내, 자연어)

여덟 가지 writing persona를 Claude Code / Codex / Gemini 세션에서 자연어로 호출합니다. 별도 커맨드 없이 입력 내용에 따라 스킬이 적절한 persona로 라우팅합니다.

Researcher — in your Claude session, say:
ai session
autoresearch how does the LLM Wiki pattern compare to Zettelkasten?
질문을 주장 단위로 분해하고, 주장별 최대 3라운드 검색 후 confidence 태그를 부여하며, synthesis 초안을 작성해 저장 전에 확인을 요청합니다. → wiki/syntheses/<slug>.md
Fact-checker — in your Claude session, say:
ai session
fact-check wiki/concepts/llm-wiki.md
초안을 원자 단위 주장으로 분해해 웹 검색으로 검증하고, 판정 표(supported·contradicted·partial·unverifiable)를 <page>.factcheck.md에 작성합니다.
Curator — in your Claude session, say:
ai session
curate my wiki — what pages are most in need of attention?
공백, 고립 페이지, 구조적 취약점을 검토하고 유지 관리 계획을 제안합니다(세션 내).
Wiki-auditor — in your Claude session, say:
ai session
check my wiki for contradictions
build a glossary for my vault
전체 일관성 검사를 실행합니다. 모순, 용어 표류, 오래된 주장을 점검합니다.
★ 공통 모델 — 모든 persona는 제안 → 확인 → 실행을 따릅니다. 파일을 읽고, 제안 초안을 작성하고, 무엇이 변경될지 보여준 다음 작성합니다. 전체 목록(translator·polisher·summarizer·scaffolder 포함)은 아래 레퍼런스에 있습니다.

마무리 / 다음 단계

전체 6부 레퍼런스는 영어·한국어 튜토리얼에 있습니다. 13개 CLI 서브커맨드는 결정론 ops를, 추론 작업은 AI 세션을 사용합니다.

terminal
